## الأوامر التي شُغّلت (كما هي)
```bash
python3 --version
python3 -m venv wameed-env
curl -fsSL https://bootstrap.pypa.io/get-pip.py -o /tmp/wameed-get-pip.py
./wameed-env/bin/python /tmp/wameed-get-pip.py
rm -f /tmp/wameed-get-pip.py
./wameed-env/bin/python -m pip install --upgrade pip
./wameed-env/bin/python -m pip install torch --index-url https://download.pytorch.org/whl/cpu
./wameed-env/bin/python -c "import torch; print(torch.__version__); print(torch.cuda.is_available())"
./wameed-env/bin/python -m pip install datasets pandas psutil
. wameed-env/bin/activate && which python
. wameed-env/bin/activate && python check_env.py
```

## الأرقام الحقيقية
- إصدار Python: 3.12.3
- إصدار PyTorch: 2.12.0+cpu
- CUDA متاح؟: False
- عدد الأنوية: 4
- الرامة المتاحة وقت الاختبار: 2338.27 ميجابايت
- إصدارات المكتبات: datasets 4.8.5، pandas 3.0.3، psutil 7.2.2
- اختبار CSV: صف واحد، وثمانية أعمدة صحيحة.

## المشاكل التي ظهرت وحلولها
- فشل إنشاء `venv` كاملاً بسبب غياب `ensurepip` ورسالة النظام طلبت `python3.12-venv`. لم تكن صلاحية `sudo` متاحة بدون كلمة مرور، لذلك ثُبّت `pip` داخل البيئة عبر `get-pip.py`.
- لم يُنشأ ملف `activate` تلقائياً بسبب نفس المشكلة، لذلك أُضيف ملف تفعيل بسيط يضبط `VIRTUAL_ENV` و`PATH`.
- في أول نسخة من `check_env.py` كانت دالة `main()` موجودة لكنها غير مستدعاة، فخرج السكربت بلا طباعة. أُضيف شرط `if __name__ == "__main__": main()`.

## القرارات والمقايضات
- استُخدم PyTorch من رابط CPU الرسمي لضمان عدم تثبيت CUDA.
- استُخدم CSV خفيف بدل أدوات تتبّع ثقيلة لتقليل استهلاك الرامة.
- استُخدم `get-pip.py` كحل داخل حساب المستخدم لأن تثبيت حزمة النظام `python3.12-venv` احتاج `sudo`.
